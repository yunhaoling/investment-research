## 一、企业级 B2B SaaS 护城河的核心逻辑

企业级 B2B SaaS 的护城河与消费品完全不同。分析前需建立以下基础认知：

| 维度 | 企业级 B2B SaaS 特征 |
|------|----------------------|
| **核心护城河** | 数据引力 / 迁移成本 / 生态锁定 |
| **关键用户指标** | NRR / RPO / $1M+ 客户数 |
| **收入模式** | 消费制（用量计费）或订阅制（固定费用），或混合 |
| **竞争转换成本** | 极高（数据迁移以工程月计，涉及组织流程重建）|
| **增长核心信号** | 存量客户消费/合同扩张（NRR > 100%）|
| **主要风险来源** | 云厂商内置化 / 竞品分流特定工作负载 / AI 架构重构 |
| **情绪信号渠道** | G2 / Gartner Peer Insights / 开发者社区 / 技术博客 |

## 二、Agent 执行前置检查（每次分析必须完成）

在采集任何数据前，Agent 必须先完成以下两项前置检查。跳过此步骤会导致分析基于过期数据或口径错误的数字。

---

### 2.1 当前日期与最新财季确认

**第一步：确认当前日期**

```
今天的日期是：[从系统获取，格式 YYYY-MM-DD]
```

**第二步：推算目标公司的最新已披露财季**

根据当前日期和公司财年截止月份，推算出最近一个**已发布**的季报：

```
公司财年截止月份：[例：1月31日 / 6月30日 / 12月31日]
当前日期：[YYYY-MM-DD]
最新已披露财季：[例：FY2026 Q4 / Q3 FY2026]
财报发布日期：[从 IR 页面或 EDGAR 确认]
```

> ⚠️ **防错规则**：若无法从搜索结果中找到发布日期确认，**不得假设**某季报已发布。应通过以下方式验证：
> ```
> web_search: "[公司名] earnings date [季度] [年份]"
> web_fetch: [公司 IR 日历页面 URL]
> ```

**第三步：搜索时强制带入年份和季度**

所有财务数据搜索必须包含具体季度和年份，禁止使用模糊表述：

```
✅ 正确：web_search: "[公司名] Q4 FY2026 earnings results"
❌ 错误：web_search: "[公司名] latest earnings"  ← 可能抓到上一季度或旧文章
```

---

### 2.2 数据校验规范

采集到数据后，在填入指标之前，必须执行以下校验步骤：

#### 校验 1：来源优先级确认

每个核心指标必须来自以下**一级来源**，不得仅依赖二级来源（新闻报道、分析师摘要等）：

| 优先级 | 来源类型 | 说明 |
|--------|----------|------|
| P1（必用） | SEC EDGAR 官方文件（8-K / 10-Q / 10-K） | 法律约束，数字最权威 |
| P1（必用） | 公司官方财报新闻稿（Press Release） | 含 Non-GAAP 指标定义 |
| P2（补充） | 财报电话会议文字记录（Earnings Call Transcript） | 管理层表述和指引 |
| P3（辅助） | 分析师报告 / 财经媒体 | 仅用于交叉参考，不得作为唯一来源 |

#### 校验 2：GAAP vs Non-GAAP 口径标注

所有指标填入时必须标注口径，禁止混用：

```yaml
product_gross_margin:
  value: 76.2%
  basis: Non-GAAP          # ← 必须标注
  source: Press Release p.4

operating_margin:
  value: -3.1%
  basis: GAAP               # ← GAAP 和 Non-GAAP 差异可能超过 20ppt，混用会导致严重误读
  source: 10-Q filing
```

#### 校验 3：异常值触发复核

当采集到的数据满足以下任一条件时，**必须暂停并返回一级来源复核**，不得直接使用：

| 异常触发条件 | 处理方式 |
|------------|---------|
| NRR 单季变化 > ±5ppt | 返回财报原文，确认是否有口径调整说明 |
| 产品营收增速 QoQ 变化 > ±10ppt | 检查是否存在并购、剥离、会计准则变更 |
| $1M+ 客户数单季下降 | 检查公司是否调整了客户计算方式（常见于客户合并） |
| RPO 大幅下降（> 15% QoQ） | 确认是否为正常确认消耗，还是合同取消 |
| 指标与二级来源报道出入 > 2ppt | 以一级来源为准，记录差异原因 |

#### 校验 4：时间维度一致性

进行 YoY / QoQ 比较时，确认对比季度的财年定义一致：

```
⚠️ 常见错误：公司在某财年调整了财年截止日期，导致 Q1 FY2025 ≠ Q1 FY2024 的时间长度。
处理方式：若发现财年调整，在指标旁注明「不可直接比较，已调整」。
```

#### 校验 5：采集完成确认清单

每季度数据采集结束后，执行以下确认：

```
□ 所有 P1 指标均来自一级来源（EDGAR 或官方 Press Release）
□ GAAP / Non-GAAP 口径已全部标注
□ 异常值已复核，原因已记录
□ 对比基期（上季 / 上年同期）财季确认无误
□ 指引数字已与电话会议文字记录交叉确认
```

## 三、五类护城河强度评估框架

每次分析时，对以下五类护城河逐一评估，标注强度（★★★★★ 到 ★）和信号颜色（green / amber / red）。

---

### 2.1 迁移成本护城河（Switching Costs）

**主要适用**：数据平台、ERP、CRM、安全工具、工作流平台

**强度驱动因素**：
- 数据体量（TB/PB 级 → 迁移成本呈超线性增长）
- 工作流深度（上下游管道数、依赖应用数量）
- 团队技能绑定（员工培训与认证积累）
- API 和集成的覆盖广度

**核心可观测指标**：
```yaml
nrr: ~                        # 净收入留存率（迁移成本的最直接体现）
gross_retention_rate: ~        # 毛留存率（排除扩张，反映纯流失）
churn_rate: ~                  # 消费额口径流失率（非客户数口径）
large_customer_growth: ~       # $1M+ 客户增速（深度依赖的代理指标）
rpq_vs_rev_growth: ~           # RPO 增速 vs 产品营收增速（合同锁定趋势）
contract_length_trend: ~       # 平均合同期限变化趋势
```

**行业健康基准**（按成熟度分层）：

| 公司阶段 | NRR 优秀 | NRR 正常 | 预警线 |
|----------|----------|----------|--------|
| 早期增长（ARR < $500M） | ≥ 130% | 115–130% | < 115% |
| 成长期（ARR $500M–$3B） | ≥ 120% | 110–120% | < 110% |
| 成熟期（ARR > $3B） | ≥ 115% | 105–115% | < 105% |

---

### 2.2 网络效应护城河（Network Effects）

**主要适用**：数据市场、协作平台、支付网络、API 平台

**两种形态**：
- **直接网络效应**：用户增加 → 每位用户获得的价值提升（如协作工具、通信平台）
- **数据网络效应**：数据/用户增加 → 模型/匹配质量提升 → 吸引更多用户（如数据市场、AI 平台）

**核心可观测指标**：
```yaml
marketplace_listings: ~         # 数据集/产品/插件数量
cross_org_connections: ~        # 跨企业数据共享或协作关系数
platform_customer_coverage: ~   # 行业渗透率代理指标（如 Forbes G2000 覆盖率）
partner_ecosystem_size: ~       # 合作伙伴与集成数量
```

> **盲区警告**：网络效应指标往往是公司最不愿意主动披露的。若财报中缺失，应在电话会议中主动追问，并结合第三方市场数据交叉验证。

---

### 2.3 平台生态锁定（Ecosystem Lock-in）

**主要适用**：云平台、开发者工具、数据平台、低代码/无代码平台

**强度驱动因素**：
- 工作负载类别多样性（每增加一类，迁移成本呈乘数增长）
- ISV/应用生态（第三方在平台上构建了多少产品）
- 开发者技能绑定（专有 API / SDK / 语言）

**核心可观测指标**：
```yaml
workload_categories_per_customer: ~  # 每客户使用的工作负载类别数
native_apps_count: ~                  # 平台上的原生应用数
isv_partner_count: ~                  # 独立软件集成合作伙伴数
developer_community_size: ~           # GitHub Stars / Stack Overflow 活跃度
```

**深化 vs 分散化判断**：
- **深化信号**：每客户工作负载类别数增加；$1M+ 客户增速快于总客户增速
- **分散化信号**：客户将特定工作负载迁移到专项竞品（如 AI 工作负载分流到原生 AI 平台）

---

### 2.4 规模经济（Scale Economics）

**主要适用**：云基础设施、数据平台、AI 训练平台

B2B 规模经济的核心体现在**单位基础设施成本递减**，而非用户边际成本递减。

**核心可观测指标**：
```yaml
product_gross_margin: ~           # 产品毛利率（排除专业服务）
infrastructure_cost_per_unit: ~   # 单位计算/存储成本趋势
fcf_margin: ~                     # 自由现金流利润率（规模杠杆的综合体现）
r_and_d_as_pct_revenue: ~         # R&D 占收入比（是否随规模递减）
s_and_m_efficiency: ~             # 销售费用效率（Magic Number 或 CAC Payback）
```

**行业健康基准**：

| 指标 | 优秀 | 正常 | 警报 |
|------|------|------|------|
| 产品毛利率 | ≥ 75% | 70–75% | < 70% |
| FCF Margin（成熟期） | ≥ 25% | 15–25% | < 15% |
| Non-GAAP 运营利润率 | ≥ 20% | 10–20% | < 5% |

> 若产品毛利率 YoY 下降 > 2ppt，需追问原因：是基础设施成本上升？新产品拖累？还是竞争引发定价压力？

---

### 2.5 品牌与信任护城河（Brand / Trust）

B2B 品牌护城河的核心不是消费者认知，而是企业 CTO / CDO 心目中的「品类默认选择」地位，以及对安全和合规的信任。

**核心可观测指标**：
```yaml
gartner_magic_quadrant_position: ~  # Gartner 魔力象限位置（Leader / Visionary / Challenger）
g2_category_ranking: ~               # G2 企业评分排名（相对竞品）
analyst_report_citations: ~          # 分析师报告引用频率
enterprise_security_certifications: ~ # SOC2 / ISO 27001 / FedRAMP 等认证状态
competitive_win_rate: ~              # 竞争性 RFP 赢单率（可从管理层表述中推断）
```

> 魔力象限位置变化（Leader → Visionary）是护城河受损的早期公开信号，应持续追踪。

## 四、MODULE 1：财务数据采集

**目标**：从 SEC EDGAR 或公司 IR 页面获取最新季度所有核心指标。

### 1.1 定位最新财报

```
web_search: "[公司名] quarterly earnings [季度] site:sec.gov"
web_search: "[公司名] investor relations earnings [季度] [年份]"
```

**注意**：采集前确认公司财年截止日期（例：1月31日 / 6月30日 / 12月31日），避免季度对照错误。

---

### 1.2 消费制 vs 订阅制采集差异

**消费制（用量计费）采集重点**：
```yaml
product_revenue: ~            # 产品营收（非总营收，排除专业服务）
product_revenue_yoy: ~        # YoY 增速
professional_services_rev: ~  # 专业服务（通常低毛利，单独追踪）
consumption_trends: ~         # 管理层对消费节奏的描述
```

**订阅制（传统 SaaS）采集重点**：
```yaml
arr: ~                   # 年化经常性收入
arr_growth_yoy: ~
new_arr: ~               # 新增 ARR
expansion_arr: ~         # 扩张 ARR（现有客户增加）
churn_arr: ~             # 流失 ARR
net_new_arr: ~           # 净新增 ARR（扩张 - 流失）
```

---

### 1.3 B2B SaaS 核心采集字段（全类型通用）

```yaml
# ── 留存与扩张（最重要）──────────────────────────────
nrr: ~                        # 净收入留存率
gross_retention_rate: ~        # 毛留存率

# ── 前瞻性指标 ──────────────────────────────────────
rpo: ~                         # 剩余履约义务（已签未确认收入）
rpo_growth_yoy: ~
current_rpo: ~                 # 12个月内将确认的 RPO
crpo_growth_yoy: ~

# ── 客户质量 ────────────────────────────────────────
customers_1m_plus: ~           # $1M+ TTM 营收客户数
customers_1m_yoy: ~
customers_10m_plus: ~          # $10M+ 客户数（战略依赖指标）
total_customers: ~
net_new_customers_qoq: ~
enterprise_penetration: ~      # 如 Forbes G2000 / Fortune 500 覆盖率

# ── 财务质量 ────────────────────────────────────────
product_gross_margin_non_gaap: ~
operating_margin_non_gaap: ~
fcf_margin: ~
cash_and_investments: ~
net_revenue_per_employee: ~    # 人均营收效率

# ── 增长动能 ────────────────────────────────────────
ai_or_new_product_accounts: ~  # AI / 新产品采用账户数
ai_revenue_run_rate: ~         # AI 相关收入规模（如适用）
new_product_adoption_rate: ~   # 新功能/产品渗透率
```

---

### 1.4 消费制的特殊解读规则

消费制收入的核心特点是**季度内实时可见性低于订阅制**。

| 信号 | 解读 |
|------|------|
| 产品营收**超出指引** | 存量客户消费加速 → 护城河加深信号 |
| 产品营收**低于指引** | 消费放缓 → 追问：宏观环境？竞品分流？效率提升导致同量工作消耗减少？|
| RPO 增速**持续 >** 产品营收增速 | 客户签更长期合同 → 锁定加深的领先指标 |
| 指引区间**明显收窄** | 管理层对消费可见性的信心增强，通常是正面信号 |

## 五、MODULE 2：护城河信号判断

### 4.1 迁移成本护城河判断矩阵

| 指标组合 | 判断 | 颜色 |
|----------|------|------|
| NRR ≥ 120% + $1M+ 客户增速 > 20% | 护城河强化中 | 🟢 green |
| NRR 110–120% + $1M+ 客户稳定 | 护城河稳定 | 🟡 amber |
| NRR < 110% 连续两季 | 护城河受损预警 | 🔴 red |
| RPO 增速持续 > 产品营收增速 | 锁定加深（正向领先信号）| 🟢 green |
| 产品营收加速 + NRR 上升同步出现 | 双重验证 | 🟢 green（加权）|
| 大客户流失 + NRR 下滑 | 结构性风险 | 🔴 red（升级）|

---

### 4.2 AI 对现有护城河的影响矩阵

| 场景 | 对护城河的影响方向 |
|------|-------------------|
| AI 应用需要企业数据 → 在平台内训练/推理 | 加深数据引力 ✅ |
| AI 提升工作效率 → 同等任务消耗减少 | 短期消费承压，长期可能拓展使用范围 ⚠️ |
| AI 原生竞争者具备更强原生架构吸引力 | 分流特定工作负载 ❌ |
| 公司 AI 功能实现货币化 | 扩展 NRR 基础，新增收入来源 ✅ |
| AI 大幅降低迁移工具的技术门槛 | 削弱迁移成本护城河 ❌ |

---

### 4.3 护城河综合评分卡（每季度更新）

| 护城河类型 | 当前强度 | 较上季变化 | 核心观测依据 | 颜色 |
|------------|----------|------------|-------------|------|
| 迁移成本 | ★★★★☆ | → | NRR / $1M+ / RPO | 🟡 |
| 网络效应 | ★★★☆☆ | → | 生态连接数 / 市场数据 | 🟡 |
| 平台生态 | ★★★★☆ | ↑ | 工作负载类别 / ISV 数 | 🟢 |
| 规模经济 | ★★★☆☆ | → | 毛利率 / FCF Margin | 🟡 |
| 品牌信任 | ★★★★★ | → | Gartner 位置 / G2 排名 | 🟢 |

> 护城河评级**仅在以下条件触发时更新**：NRR 变化超过 2ppt；$1M+ 客户增速明显偏离近 3 季均值；AI 采用数据出现量级变化；竞争格局重大变化；管理层在电话会议中明确表达战略转变。

## 六、MODULE 3：情绪与市场信号采集

> B2B 企业没有 App Store 评分，需要通过以下替代渠道获取用户情绪信号。

### 5.1 企业用户评价平台

| 平台 | 采集内容 | URL 格式 | 建议频率 |
|------|----------|----------|---------|
| G2 | 综合评分、功能维度评分、竞品对比 | `g2.com/products/[公司名]/reviews` | 每季度 |
| Gartner Peer Insights | 企业级评分、魔力象限验证 | `gartner.com/reviews/market/[类别]/vendor/[公司名]` | 每半年 |
| TrustRadius | 详细功能使用评价 | `trustradius.com/products/[公司名]/reviews` | 每半年 |

---

### 5.2 开发者与从业者社区信号

**通用搜索模板**（替换 `[公司名]` 和 `[竞品名]`）：

```
web_search: "[公司名] vs [竞品名] [年份] site:reddit.com"
web_search: "[公司名] review engineer stackoverflow"
web_search: "leaving [公司名] switching [竞品名] engineer"
web_search: "[公司名] pricing cost complaints"
```

**关键词监控分类**：
- ✅ **正面**：`[产品] performance` / `[功能] great` / `best [品类] tool`
- ⚠️ **中性警示**：`[公司名] expensive` / `[公司名] complex`
- ❌ **负面警报**：`migrating from [公司名]` / `[竞品] better than [公司名]`
- 🚨 **紧急警报**：`[公司名] outage` / `[公司名] security breach` / `[公司名] pricing increase`

---

### 5.3 分析师生态信号

```
web_search: "Gartner Magic Quadrant [品类] [年份]"
web_search: "Forrester Wave [品类] [年份]"
web_search: "IDC market share [品类] [年份]"
```

---

### 5.4 招聘信号（另类数据）

企业对某平台技能工程师的招聘量，是客户采用深度的代理指标：

```
web_search: "[公司名] engineer DBA jobs [年份]"
web_search: site:linkedin.com/jobs "[公司名] data engineer"
```

**解读规则**：若主要客户大规模招募竞品工程师而非该平台工程师，是迁移意图的早期信号。

## 七、MODULE 4：指标追踪文件更新规范

### 6.1 新增季度数据

每季度财报后更新数据追踪文件，需同步：
- 新增季度列（保持列宽定义一致）
- 更新关键 KPI 卡片数字（NRR、RPO、$1M+ 客户数、FCF Margin）
- 将指引值从当前季度滚动更新至下一季度

### 6.2 观察清单管理逻辑

每季度财报后执行以下流程：

1. 将上季度观察清单中的指标标记为「已验证」或「持续观察」
2. 更新前瞻性指引（下季度 / 下财年）
3. 重新评估 P1 / P2 / P3 优先级（上季度 P3 可能在新季度升为 P1）
4. **升级规则**：某指标连续两季度低于预期 → 自动升级为 P1 + 红色警报

### 6.3 护城河卡片更新触发条件

护城河评级和描述**仅在以下情况触发更新**，避免噪音驱动的频繁变动：

| 触发条件 | 说明 |
|----------|------|
| NRR 变化 > ±2ppt | 正向或负向均触发 |
| $1M+ 客户增速偏离近 3 季均值 > 5ppt | 加速或放缓均触发 |
| AI / 新产品采用出现量级变化 | 如账户数 QoQ 增速突破 30% |
| 竞争格局重大变化 | 主要竞品新产品发布、大客户公开迁移案例 |
| 管理层明确表达战略转变 | 电话会议中明确表述，非推断 |

## 八、企业级 B2B SaaS 通用指标阈值参考

以下阈值基于 2024–2026 年云软件行业基准，**按公司成熟度分层**：

| 指标 | 优秀 | 正常 | 警报 |
|------|------|------|------|
| NRR（成长期） | ≥ 120% | 110–120% | < 110% |
| 毛留存率 | ≥ 95% | 90–95% | < 90% |
| 产品毛利率 | ≥ 75% | 70–75% | < 70% |
| FCF Margin（成熟期） | ≥ 25% | 15–25% | < 15% |
| RPO / TTM 营收比 | ≥ 2.0x | 1.5–2.0x | < 1.5x |
| $1M+ 客户占总营收 | ≥ 70% | 60–70% | < 60% |
| Non-GAAP 运营利润率 | ≥ 20% | 10–20% | < 5%（成熟期）|
| 产品营收增速 | ≥ 25% | 15–25% | < 15%（成熟期）|

> **重要提示**：同一指标对早期公司（ARR < $500M）和成熟公司（ARR > $3B）的含义截然不同。使用阈值前需先判断公司所处阶段，并与**同类阶段**的可比公司对照，而非使用绝对值横向比较。

## 九、MODULE 5：估值分析框架（PE / PS / DCF × Bear / Base / Bull）

估值分析分三层递进：**方法论层**（PE / PS / DCF 三种工具）、**情景层**（Bear / Base / Bull 三种假设）、**比较层**（历史纵向 + 行业横向）。三层叠加后得出综合估值区间判断。

### 9.2 方法一：PS 倍数估值（Price-to-Sales）

PS 是 SaaS 公司最常用的相对估值工具，适用于尚未盈利或盈利不稳定的成长期公司。

#### 核心公式

```
PS（TTM） = 市值 ÷ TTM 产品营收
EV/NTM Revenue = EV ÷ NTM 营收预期
```

#### Bear / Base / Bull 情景假设

| 情景 | NTM 营收增速假设 | 合理 PS 倍数区间 | 对应目标市值 | 对应目标价 |
|------|----------------|----------------|------------|---------|
| 🔴 Bear | [填入，如 12%] | [填入，如 8–10x] | [计算] | [计算] |
| 🟡 Base | [填入，如 20%] | [填入，如 12–15x] | [计算] | [计算] |
| 🟢 Bull | [填入，如 28%] | [填入，如 18–22x] | [计算] | [计算] |

**Bear / Base / Bull 情景差异驱动因素**（每次分析需明确填写）：

```yaml
bear_scenario_drivers:
  - NRR 持续下滑至 [X%]，存量客户消费增长放缓
  - 竞品分流 [具体工作负载类型]，导致增速低于指引
  - 宏观环境收紧，企业 IT 预算削减

base_scenario_drivers:
  - NRR 维持在 [X%]，与近 4 季均值一致
  - 新产品（AI / 平台扩展）贡献 [X%] 增量营收
  - 宏观环境平稳，不存在超预期上下行

bull_scenario_drivers:
  - NRR 回升至 [X%]，AI 工作负载带动消费加速
  - $1M+ 客户增速重新超过 25%
  - 新产品渗透率超预期，提前贡献规模营收
```

#### PS 倍数合理性判断规则

```
Rule 1：EV/NTM Revenue 倍数 ÷ NTM 营收增速 = PEG 类比值（Rule of 40 调整版）
        → 若该比值 > 1.5，估值偏高；< 0.8，估值偏低（相对历史均值）

Rule 2：PS 倍数应随 FCF Margin 提升而提升
        → FCF Margin 每提升 5ppt，合理 PS 倍数上移约 1–2x（行业经验值）

Rule 3：消费制公司的 PS 倍数天然低于订阅制
        → 相同增速下，消费制公司给予 10–15% 折价是合理的
```

### 9.3 方法二：PE / FCF 倍数估值（盈利能力估值）

适用于已实现正向 FCF 或 Non-GAAP 盈利的公司。对成熟期 SaaS 公司更具锚定意义。

#### 核心公式

```
P/FCF（TTM） = 市值 ÷ TTM FCF
EV/NTM FCF  = EV ÷ NTM FCF 预期
Non-GAAP P/E = 市值 ÷ TTM Non-GAAP 净利润
```

#### Bear / Base / Bull 情景假设

| 情景 | NTM FCF Margin 假设 | NTM FCF 绝对值 | 合理 P/FCF 倍数 | 对应目标价 |
|------|-------------------|--------------|--------------|---------|
| 🔴 Bear | [填入，如 15%] | [计算] | [填入，如 25–30x] | [计算] |
| 🟡 Base | [填入，如 22%] | [计算] | [填入，如 35–40x] | [计算] |
| 🟢 Bull | [填入，如 28%] | [计算] | [填入，如 45–50x] | [计算] |

**P/FCF 倍数合理性判断规则**：

```
Rule 1：高成长 SaaS（营收增速 > 20%）P/FCF 通常在 40–60x
        成熟 SaaS（营收增速 10–20%）P/FCF 通常在 25–40x
        低增长 SaaS（营收增速 < 10%）P/FCF 通常在 15–25x

Rule 2：P/FCF 与 FCF 增速的比值（FCF PEG）
        → FCF PEG < 1.0 = 相对低估；> 2.0 = 相对高估

Rule 3：若公司 FCF 仍为负，此方法不适用，退回 PS 估值为主
```

---

### 9.4 方法三：DCF 估值（现金流折现）

DCF 是估值的「锚」，用于检验 PS / PE 隐含的市场预期是否合理。SaaS 公司 DCF 的核心变量是**终局 FCF Margin** 和**长期增速**。

#### 关键假设填写模板

```yaml
dcf_assumptions:
  # ── 阶段一：高增长期（第 1–5 年）────────────────────
  phase1_revenue_cagr:
    bear: ~      # 如 12%
    base: ~      # 如 20%
    bull: ~      # 如 28%
  phase1_fcf_margin_exit:
    bear: ~      # 如 18%
    base: ~      # 如 25%
    bull: ~      # 如 32%

  # ── 阶段二：过渡期（第 6–10 年）─────────────────────
  phase2_revenue_cagr:
    bear: ~      # 如 6%
    base: ~      # 如 12%
    bull: ~      # 如 18%
  phase2_fcf_margin_exit:
    bear: ~      # 如 22%
    base: ~      # 如 28%
    bull: ~      # 如 35%

  # ── 阶段三：永续期 ───────────────────────────────────
  terminal_growth_rate:
    bear: ~      # 如 2.5%
    base: ~      # 如 3.5%
    bull: ~      # 如 4.5%
  terminal_fcf_margin:
    bear: ~      # 如 25%
    base: ~      # 如 30%
    bull: ~      # 如 38%

  # ── 折现率（WACC）────────────────────────────────────
  wacc:
    bear: ~      # 如 11%（利率高企 / 风险溢价高）
    base: ~      # 如 9%
    bull: ~      # 如 8%（低利率环境 / 高质量资产溢价）
```

#### DCF 敏感性分析矩阵（Base 情景下）

对终局 FCF Margin 和 WACC 做双维度敏感性，快速识别估值的核心驱动变量：

```
                  WACC
终局 FCF Margin   7%     9%     11%    13%
      20%        [价格]  [价格]  [价格]  [价格]
      25%        [价格]  [价格]  ← 基准  [价格]
      30%        [价格]  [价格]  [价格]  [价格]
      35%        [价格]  [价格]  [价格]  [价格]
```

> **DCF 使用原则**：DCF 结果的绝对数字价值有限，更有价值的是「**市场当前隐含了什么假设**」的反推（Reverse DCF）。若当前股价对应的隐含增速远超历史基准，即为高估信号。

#### Reverse DCF：解读市场隐含预期

```
当前股价 → 反推隐含的 5 年营收 CAGR 和终局 FCF Margin
→ 与公司历史增速、行业基准、管理层指引对比
→ 判断市场预期是否合理
```

### 9.7 行业横向比较（同类公司维度）

**目标**：判断公司相对同类可比公司是否存在溢价或折价，并理解溢价/折价的基本面依据。

#### 可比公司选取原则

选取 4–6 家可比公司，满足以下条件：
1. **同一行业细分**（数据平台 / 安全 / 协作 / ERP 等，不跨大类比较）
2. **相近成熟度**（ARR 规模相近，避免拿早期公司和成熟公司直接比较）
3. **相似收入模式**（消费制 vs 消费制；订阅制 vs 订阅制）

**数据来源**：
```

```

#### 横向溢折价合理性判断

溢价合理的条件（需同时满足 2 条以上）：
- NRR 高于同类中位数 > 5ppt
- 营收增速高于同类中位数 > 5ppt
- FCF Margin 改善趋势更陡（YoY 改善幅度更大）
- 护城河评分卡总分高于同类（生态锁定更深）
- TAM 扩张空间更大，尚处市场渗透早期

折价需警惕的信号：
- NRR 低于同类中位数且持续下滑
- 增速放缓但估值折价未充分体现
- 竞品在 Gartner MQ 中持续超越

## 十、如何将此模板适配到具体公司

将此模板适配到具体 B2B SaaS 公司时，需调整以下五个维度：

1. **财年日期**：确认财年截止月份，避免季度对照错误
2. **收入模式**：消费制 vs 订阅制 → 调整采集字段（产品营收 vs ARR）
3. **护城河权重**：根据行业主导护城河调整优先级（ERP = 迁移成本极高；协作工具 = 网络效应；安全工具 = 品牌信任）
4. **行业基准**：安全软件 NRR 通常高于数据平台；基础设施软件毛利率低于纯 SaaS
5. **竞争对手**：替换竞争格局分析中的对手，并更新情绪监控关键词

**适配成功的验证标准**：五类护城河中，每一类都能找到至少一个可从财报中直接提取的可观测指标。
